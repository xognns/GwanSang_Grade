## User Scenarios & Testing *(mandatory)*

### User Story 1 - 사진으로 성적 조회 (Priority: P1)

사용자는 이름과 얼굴 사진을 제출해 재미용 성적 결과를 즉시 확인할 수 있어야 한다.

**Why this priority**: 서비스의 핵심 가치는 사진 제출부터 결과 확인까지의 단일 흐름에 있다.

**Independent Test**: 사용자가 메인 화면에서 이름을 입력하고 단일 얼굴 사진을 올린 뒤
`성적 조회 시작`을 눌렀을 때 결과 화면이 완성된 상태로 표시되면 이 스토리는 독립적으로
검증된다.

**Acceptance Scenarios**:

1. **Given** 사용자가 메인 화면에 있고 이름과 유효한 단일 얼굴 사진을 준비했을 때,
   **When** 성적 조회를 시작하면, **Then** 결과 화면에 이름, 업로드한 사진, `0.0 ~ 4.5`
   범위의 점수, 등급, 한줄 코멘트, 칭호가 표시된다.
2. **Given** 사용자가 결과를 조회했을 때, **When** 결과 화면을 확인하면, **Then** 서비스가
   재미용이며 실제 능력과 무관하다는 안내를 함께 확인할 수 있다.

---

### User Story 2 - 잘못된 사진에 대한 안내 (Priority: P2)

사용자는 분석이 불가능한 사진을 제출했을 때 이유를 이해하고 다시 시도할 수 있어야 한다.

**Why this priority**: 실패 상황을 이해하지 못하면 사용자는 핵심 흐름을 끝까지 경험하지 못한다.

**Independent Test**: 사용자가 얼굴이 없거나 여러 얼굴이 있는 사진, 또는 지원하지 않는
파일을 제출했을 때 결과 대신 구체적인 오류 안내와 재시도 경로를 보면 이 스토리는
독립적으로 검증된다.

**Acceptance Scenarios**:

1. **Given** 사용자가 얼굴이 없거나 여러 얼굴이 포함된 사진을 제출했을 때, **When** 분석을
   시작하면, **Then** 서비스는 왜 실패했는지 설명하는 메시지를 보여주고 결과 화면 대신
   재시도를 유도한다.
2. **Given** 사용자가 지원하지 않는 파일 형식이나 허용 크기를 넘는 파일을 제출했을 때,
   **When** 제출을 시도하면, **Then** 서비스는 즉시 오류 사유를 알려주고 올바른 입력 조건을
   안내한다.

---

### User Story 3 - 결과 다시 보기와 복귀 (Priority: P3)

사용자는 결과를 본 뒤 같은 사진으로 다시 조회하거나 메인 화면으로 돌아가 다음 시도를
이어갈 수 있어야 한다.

**Why this priority**: 이 서비스는 재미와 반복 사용성이 중요하므로 결과 확인 후의 다음 행동이
막히지 않아야 한다.

**Independent Test**: 사용자가 결과 화면에서 `다시 조회`와 `메인 화면으로 돌아가기`를 각각
실행했을 때 기대한 흐름으로 이동하면 이 스토리는 독립적으로 검증된다.

**Acceptance Scenarios**:

1. **Given** 사용자가 결과 화면에 있을 때, **When** `다시 조회`를 누르면, **Then** 같은 입력을
   기준으로 새로운 결과를 다시 확인할 수 있다.
2. **Given** 사용자가 결과 화면에 있을 때, **When** `메인 화면으로 돌아가기`를 누르면,
   **Then** 다음 제출을 시작할 수 있는 초기 진입 화면으로 돌아간다.

### Edge Cases

- 이름은 입력했지만 사진이 없을 때 조회를 시작할 수 없어야 한다.
- 사진은 올렸지만 이름이 비어 있을 때 조회를 시작할 수 없어야 한다.
- 얼굴이 전혀 감지되지 않으면 결과 대신 실패 이유를 알려야 한다.
- 두 명 이상이 포함된 사진이면 단일 얼굴만 허용된다는 점을 알려야 한다.
- 분석 도중 일시적 오류가 발생해도 사용자는 다시 시도하거나 메인으로 복귀할 수 있어야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 분석 시작 전에 사용자가 결과 표시용 이름을 입력할 수 있게 해야 한다.
- **FR-002**: 시스템은 한 번의 분석 요청마다 얼굴 이미지 1장만 제출받아야 한다.
- **FR-003**: 시스템은 이름과 유효한 이미지가 모두 준비되기 전에는 분석을 시작할 수 없게 해야 한다.
- **FR-004**: 시스템은 지원하지 않는 파일 형식, 허용 크기 초과 파일, 얼굴이 없는 이미지, 여러 얼굴이 있는 이미지에 대해 사용자 안내와 함께 거절해야 한다.
- **FR-005**: 시스템은 유효한 제출마다 `0.0 ~ 4.5` 범위로 정규화된 재미용 결과 1건을 반환해야 한다.
- **FR-006**: 시스템은 유효한 결과마다 제출한 이름, 제출한 사진, 점수, 등급, 짧은 코멘트, 2개 이상의 칭호를 표시해야 한다.
- **FR-007**: 시스템은 결과가 재미용이며 실제 학업 평가가 아니라는 점을 명확히 안내해야 한다.
- **FR-008**: 사용자는 결과 화면에서 같은 입력으로 다시 조회할 수 있어야 한다.
- **FR-009**: 사용자는 결과 화면에서 메인 화면으로 돌아가 새로운 제출을 시작할 수 있어야 한다.
- **FR-010**: 시스템은 이 MVP 범위에서 계정 생성, 랭킹, 친구 비교, 공유 카드 생성을 요구하지 않아야 한다.

### Key Entities *(include if feature involves data)*

- **Analysis Request**: 사용자가 입력한 이름과 단일 얼굴 이미지 제출 정보를 담는 요청 단위다.
- **Analysis Result**: 점수, 등급, 코멘트, 세부 표현 요소, 칭호를 포함한 결과 묶음이다.
- **Reference Baseline**: 결과 점수 범위를 해석할 때 기준점 역할을 하는 내부 기준 정보다.
- **Validation Error**: 사용자 입력이 분석 조건을 충족하지 못할 때 표시할 오류 상태와 안내 문구다.

## Contract & Documentation Impact *(mandatory)*

- **Docs Updates**: `Docs/mvp-spec.md`를 본 spec 기준으로 정렬하고, 분석 요청/응답/오류 사례를 정리한 계약 문서를 `Docs/` 아래에 추가하거나 갱신해야 한다.
- **API Contract Changes**: 이름과 이미지 제출 계약, 결과 조회 응답 계약, 오류 응답 계약을 명확히 정의해야 한다.
- **Frontend Impact**: 메인 입력 화면, 이미지 미리보기, 결과 화면, 오류 안내, 재조회 및 복귀 동선이 필요하다.
- **Backend Impact**: 제출 검증, 기준점 기반 점수 계산, 결과 구성, 실패 사유 구분이 필요하다.
- **Separation Guarantee**: 프론트엔드는 `Docs/`에 정의된 입력과 출력 계약만 사용하고, 백엔드는 그 계약에 맞는 결과와 오류만 반환한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 유효한 제출의 95% 이상은 사용자가 이름 입력부터 결과 확인까지 1분 이내에 완료할 수 있어야 한다.
- **SC-002**: 지원하지 않는 입력 사례의 100%는 빈 화면이나 부분 결과 대신 구체적인 오류 메시지를 제공해야 한다.
- **SC-003**: 사용자 검토에서 90% 이상이 첫 화면 또는 결과 화면만 보고도 이 서비스가 재미용이라는 점을 정확히 이해해야 한다.
- **SC-004**: 유효한 결과의 100%는 점수, 등급, 코멘트, 최소 2개의 칭호, 재조회 또는 복귀 행동을 모두 포함해야 한다.

## Assumptions

- 초기 MVP는 라이브 카메라 촬영보다 이미지 업로드 경험을 우선 제공한다.
- 한 번의 제출에는 단일 얼굴만 허용한다.
- 얼굴 이미지는 즉시 결과를 생성하기 위한 범위에서만 사용하고 기본적으로 영구 저장하지 않는다.
- 로그인, 랭킹, 친구 비교, 공유 카드 생성은 초기 MVP 범위에서 제외한다.
